책벗 프로젝트 진행상황

[프로젝트 진행상황]

  1. 회사일이 바빠서 최근에 개발도 거의 못하고 블로그 글도 못썼습니다. 어차피 시작한지 얼마되지 않은 블로그라 보는 사람도 없지만, 초기에 너무 글이 없다보니 벌써 폐쇄위기에 처한 것 같아 오랜만에 글을 씁니다.
  2. 책벗, 책친구라는 뜻이라 영어로는 book buddy가 되어야 하는데 최근에 골프를 열심히 연습하면서 버디 욕심이 생겼는지 book birdie라고 태그를 써버렸네요. 어떻게 지우는지 몰라서 태그가 그대로 남았습니다. 나중에 연구해서 지워야겠습니다.

책벗 디자인

시작화면

시작은 심플하게 이렇게 로고만 띄우는 것으로 했습니다. 어차피 데이터를 읽어들이는 짧은 시간동안만 노출할 거라 용량을 최소화 하고 싶었습니다. 무료 아이콘 사이트, pngtree 등에서 판매하는 아이콘들은 CI용으로 쓸 수 없는 것이 대부분이라 책 모양 아이콘은 일러스트레이터에서 직접 그렸습니다. 메인 폰트는 로고용으로 쓸 수 있도록 라이센스가 허용된 Gmarket Sans 폰트를 이용했습니다.


도서목록

도서 목록은 그리드형(액자보기?), 리스트로 보기, 달력보기 3가지를 하단의 탭바를 통해 간편하게 이동할 수 있도록 했습니다. 맨 오른쪽 버튼은 통계보기인데 아직 통계 화면은 디자인을 못했네요. 책 추가는 FAB으로 처리했습니다.

완독/읽는중/읽을책은 그리드와 리스트에서는 배지를 달아서 표시할 수 있게하고 달력보기에는 완독한 책만 노출할 예정입니다. 달력을 위 그림과 같이 스크롤이 가능하게 만들 수 있을지는 구현하면서 생각해봐야겠습니다.

상단의 셀렉트 박스를 이용해서 검색과 필터를 동시 적용할 수 있도록 할 생각입니다.

아직도 의심스러운 건 상하단 메인컬러로 잡은 저 색깔입니다. 전문 디자이너가 아니다보니 모든 디자인이 다 어렵지만, 특히나 브랜드 컬러를 정하는게 정말 어렵습니다.


기타화면

책 데이터가 없을 때 노출할 빈책장 그림은 메인 아이콘을 이용해 만들었습니다. 나름 마음에 듭니다. 그리드와 리스트에서 책 그림을 꾹 누르면 퀵메뉴가 뜨고 빠르게 완독설정을 할 수 있도록 했습니다.

아직 설정메뉴, 로그인화면, 통계, 책 정보 등등 디자인 해야할 화면들이 많지만 디자인만 계속하다보니 물려서 코드를 하나씩 만들기 시작했습니다. Firebase는 이번에 처음 써보는데 따로 백엔드를 구성할 필요가 없다는 것도 이번에 알았네요.

아직 XD로 화면도 다 못그렸는데 진도가 너무 더디네요. 연말만 지나면 상황이 좀 괜찮아질 것 같긴한데, 빨리 완성해서 쓰고 싶습니다. 무엇보다 책벗은 절 위한 앱이니까요.

오늘도 쓸데없는 글 읽어주셔서 감사합니다.

Firebase Null Safety 적용 문제

저는 flutter 버전 2.5.3, Dart 2.14.4의 최신버전을 사용하고 있습니다. Flutter를 선택한 이유에서도 썼지만 팀원과의 호환성을 생각하지 않고 최신버전을 언제든 적용해볼 수 있다는 것이 개인개발자의 장점입니다.


하지만 문제는 팀원과의 호환 뿐만 아니라 패키지 사이의 호환성 문제도 발생한다는 점인데요. Flutter는 2.0 부터 Null Safety를 적용하기 때문에 Null Safety가 적용 안된 Firebase 패키지를 사용하면 아래와 같은 에러가 발생합니다.

해결을 위해서는 main.dart에 // @dart=2.9 를 선언하여 강제로 낮은 버전의 dart를 적용하든지 null safety가 적용된 패키지를 사용해야합니다.

Dart 패키지 2.10부터 Null safety를 사용하기 때문에 2.9이하의 버전을 사용해서 이를 해결하는 방법입니다.

하지만 버전업 주기가 짧은 게 플러터의 장점이고, 기껏 성능이 개선된 SDK를 적용했는데 Null Safety 때문에 과거로 돌아갈 수는 없습니다. 이미 코틀린에서 Null Safety를 지키며 코딩을 해왔기 때문에 어려운 개념도 아니구요. 그래서 최대한 사용하는 패키지를 Null Safety 호환 버전으로 교체하거나 Null safety를 적용할 수 있도록 migrate 하는 방법을 택하겠습니다.

Null Safety를 지원하는 Firebase 패키지 버전은 다음과 같습니다.

[아래 주소에서 확인가능합니다]

firebase_auth ^3.2.0

firebase_storage ^10.1.0

저는 예전 구글 Firebase 공식문서에 있는대로 이런 구버전을 붙여놓고 있었네요. 그러면 pubspec.yaml을 해당 버전 패키지로 수정해서 에러가 없어지는지 보겠습니다.


실행했더니 null safety가 적용되어 잘 작동합니다.


null safety가 적용안된 패키지의 migrate까지 작성해보려고 했는데 지금은 그런 패키지가 없네요. Firebase도 업데이트를 빨리 잘 해주는 것 같습니다.

Flutter를 선택한 이유

팀의 부재

저는 팀 없이 혼자 코딩 작업을 하고 있습니다. 첫 번째 모바일 어플은 Kotlin으로 만든 안드로이드 앱이었는데 iOS용으로 따로 개발하는 부담이 있었습니다. 저 같은 개인개발자나 소규모 스타트업에는 코드 하나로 Android/iOS/Web 까지 한번에 빌드가 가능한 Flutter는 매력적인 제안입니다.

혼자 작업하기 때문에 최신 버전의 Flutter를 언제든 마음대로 적용해 작업할 수 있다는 점은 장점으로 작용합니다. 팀원끼리의 호환성을 걱정할 필요가 없기 때문이죠.

빠른 레이아웃 작업

모바일앱 작업을 실제로 해보니 고난도의 기술이 필요한 앱이 아니면 레이아웃 편집에 들어가는 시간이 반 이상이었습니다. 레이아웃을 그림으로 그리는게 아니라 하나씩 코드로 만들다보니 변경한 코드를 빌드해서 확인하는 시간이 많이 걸렸는데요, 플러터의 Hot Reload는 정말 빠르고 편리하게 수정된 사항을 확인할 수 있어 작업시간이 줄어드는 것을 체감할 수 있었습니다.

편리한 비동기 코드

Kotlin으로 작업할 때 코루틴이나 LiveData를 이용한 비동기 작업도 편리하긴 했지만 사용성 측면에서 Flutter의 async/await이 더 쉬운 느낌이었습니다.


이제 개발을 시작하는 단계라 어떤 벽에 부딪힐 지는 모르겠지만 Flutter를 쓰면서 생기는 문제와 해결과정을 블로그에 정리해나갈 생각입니다.
